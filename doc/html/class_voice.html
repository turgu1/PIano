Detailed Description

Class Voice represents a sample that is currently playing through the PCM device. (Class Sample represents the static portion of a sample.) PIano, at launch time, pre-allocate 128 voices that will be associated to samples dynamically when a new sound will be required to be played. The number of pre-allocated voices can be easily changed in the Poly class definition (poly.h).

A voice (an instance of the class Voice), during its life, goes through a variety of states that enable the synchronisation of events that occurs in 4 concurrent threads:

o Midi callback processing (when a user interact with the keyboard) o PCM callback processing (when the PCM device requires new sound data) o In the Sample File Opener thread (when its time to open a sample file to read more data) o In the Samples Feeder thread (when its time to feed new sample data in the FIFO buffer)

As you may have infered, the Voice class is at the center of the activies allowing the generation of sounds to be sent to the PCM device.

The main duty of this class is to supply sound data from a single sample file to the Poly::mixer method in bundles of 256 frames. When a voice is first activated, it starts by supplying frames that have been pre-loaded at the time the Samples Library has been instantiated (at program startup or when a new library is loaded in memory). This will give (hopefully) sufficient time to background threads to open the sample file and start to feed a FIFO (First In, First Out) buffer. At the end of the pre-loaded frame buffer, the instance will then start to read data from the FIFO until the end of the sample or a request by the user to stop a note being played.

A frame contains two float values, one for each side of a stereophonic sound. The PCM device is programmed to require 44100 frames per second. A bundle will supply ~5.8ms (256 / 44100) of data to the PCM device. The maximum latency produced is then less than 12ms (A note is requested by the user: at the time the note is strucked on the keyboard, there is already a bundle being played, a second bundle is in preparation by PIano, the note strucked will be taken into account on the next bundle to be prepared).

As is often the case, a Samples Library maybe lacking some of the notes required to fullfill the entire keboard keys. This class allow for scaling of notes to reflect the pitch of the targeted note.

On the dynamics of the Voice Class

The following state diagram presents the various states through which a voice is going through when a sound is required to be played.

A voice will be playing only when its global state is ACTIVE. At startup, all voices are DORMANT and INACTIVE. When a sound is requested, a voice is activated by associating it to a sample and changing its state to OPENING and ACTIVE. Once the file asssociated with the sample has been opened, the state is changed to ALIVE. The FIFO associated with the voice is then being feeded by the SamplesFeeding thread. When a request is made to stop to play a voice, the sound is fade out, diminishing its volume to zero for the next second or the end of the sample data if it comes before. When fade out is complete, the voice states is then put as INACTIVE and CLOSING. If the voice's fade out is completed before the file has been opened (state ALIVE), it will then be put in the CLOSING state anyway.

Fade out processsing is done by the Poly::mixer() method, as it implies an optimization on computations done with the ARM NEON intrinsic instructions.
